## 2. 時系列で見る開発の流れ

大きく 4 つのフェーズに分けられる。

### フェーズ 1: 立ち上げ — コア骨格と閲覧 CLI (6/28〜6/30)

- **設計の骨格を最初に固めた**: 操作は skill (`/agent-tasks`)、閲覧は CLI (`agent-tasks`)。
  データはコードリポジトリの外 (`~/agent-tasks-store`) に置き、ブランチごとに状態がずれないようにする。
  agent 非依存の命名 (`claude-` prefix を使わない)。
- 閲覧 CLI をシェルスクリプトから **Go に移植**し、以降の土台にした。
- 並行開発の中核 (`start` = git worktree、`spawn` = 別 pane、`batch` = 直列実行) を早期に用意。
- 見える化の基礎: `list --json` / `--recent` / `version` / シェル補完 / CI / worktree フック /
  `alloc-id` (原子的採番) / TUI ビューア。
- 表示品質: East Asian Width の自前テーブルを `go-runewidth` に差し替え、
  方針を「依存ゼロ」→「**依存は最小限**」へ転換。ID の数値ソート・端末幅 truncate も整備。

### フェーズ 2: 運用を厚くする — TUI / serve / worktime / cost / claim (7/1〜7/2)

- **TUI を一気に強化**: ヘルプ overlay、セッション状態列、検索 (`/`)、PR/セッションを開く、
  クリップボードコピー、tig 風ナビ、アーカイブ操作 (初の書き込み操作)。
- **運用フィールドの拡充**: `archive`/`auto-archive`、`report`、`tracker:`、`prs:`、
  **人手タスク (`kind: human`)** — コードを触らないタスクを worktree 抜きで扱えるように。
- **serve (Web 閲覧)** を追加。LAN のスマホから一覧を見られる HTTP サーバ + Cloudflare Tunnel/Access
  での外出先公開手順。状態別セクション・project 別サブグループ・アプリディープリンク。
- **worktime (実稼働時間)** と **cost (トークン消費/概算コスト)** を計測可能に。
  session-link (task↔session_id) を軸に、リードタイム (壁時計) とは別の「実際に動いていた時間」を出す。
- **`claim` で TOCTOU を解消** (後述)。start 手順を「claim → コンフリクトチェック → worktree」に組み替え。

### フェーズ 3: herdr 全面移行 — agent 中立化 (7/2〜7/8)

このプロジェクト最大の作り直し。tmux + Claude Code 固有 hook への依存を、
エージェント対応マルチプレクサ **herdr** の socket API に一本化した。

- 設計 (0105) → クライアント層 (0106) → 各機能の移行を段階的に:
  spawn (0108) / SESSION 状態 (0109) / 自 session_id 特定 (0110) / session-rename (0111) /
  cost transcript (0112) / worktime プラグイン (0114)。
- **共存ビルドで安全に移行**: 別名バイナリ `agent-tasks-herdr` + state dir 隔離で、
  稼働中の tmux 版と並走させながら検証 (0113)。env による版ルーティング (0118) で dogfood。
- 検証を経て **本採用マージ** (0119)。命名・state dir を `agent-tasks` に復帰
  (フォーマット互換なのでデータ移行不要)。
- 効果: SESSION 状態が working だけでなく **blocked (承認/許可待ち) / idle を区別**できるように。
  `tmux capture-pane` に頼らず alt-screen でも検出。Claude 固有 hook 依存が減り agent 中立に近づいた。

### フェーズ 4: 磨き込み — 可視化とフロントエンド整備 (7/9〜7/10)

- serve に **時間帯・並列稼働ビュー** (曜日×時刻ヒートマップ → 日別スイムレーン → タスク別レーン)。
- serve フロントエンドを Go の raw 文字列 const から **外部ファイル + `//go:embed` + vitest** に分離。
- TUI のライブセッション ● 表示、pane フォーカス (`f`)、spawn (`S`)、ヘルプのバージョン表示など。
- CLI のコマンド定義をレジストリ (`commands.go`) に集約、`--flag=value` を全フラグ対応。

## 3. 設計上うまくいった点

- **操作 (skill) と閲覧 (CLI) の分離、データをコード repo の外に置く。**
  この 2 つの初期判断が全体の安定性を支えた。ブランチを切り替えてもタスク状態がぶれず、
  worktree による並行開発と自然に噛み合った。
- **agent 中立の設計。** `claude-` prefix を避けた命名と、herdr 移行での hook 依存の削減により、
  「Claude 専用ツール」ではなく「エージェント開発の器」に育てられた。
- **状態遷移を CLI のロック下に集約 (`alloc-id`/`claim`/`done`/`block`/`resume`)。**
  frontmatter の scalar キー確定を LLM の手編集に任せず、project ロック下で決定的に書き換える。
  completed_at 付け忘れ・blocked_* 消し忘れ・日付推測ミスといった「LLM が壊しやすい箇所」を
  仕組みで封じた。本文・進捗ログは保全。
- **依存最小 + 単一バイナリ。** 標準ライブラリ中心 (frontmatter パースも自前)、確立した小さな
  ライブラリだけ例外採用 (`go-runewidth`)。`go build` 一発・`//go:embed` で web アセットも同梱。
- **CHANGELOG = マージ日方式 + 並行マージ耐性。** 実装中は `[Unreleased]`、マージ時に日付セクションへ
  移す運用で、日跨ぎ PR でも日付がマージ日に一致。共有ファイルの衝突は GitHub のマージ不可検知で止め、
  main 直接 push のレースを避けた。
- **セルフホスティング (dogfooding)。** agent-tasks 自身を agent-tasks で開発した。
  一覧当日完了が 120 件中 116 件という短いリードタイムは、worktree 並行と「小さく切る」文化に加え、
  ツールを日々自分で使い倒したことで摩擦が即フィードバックされた結果でもある
  (fix 22 件の多くは dogfood 由来)。

## 4. 苦労した点・作り直した点から得た学び

- **TOCTOU (二重着手ガード) — 0065。**
  旧 start は in-progress 確定が worktree 作成の *後* で、着手指示〜確定の窓が長く、
  並行 start が互いを観測できず素通りしていた。`claim` をチェックより前に置き、
  窓を「ロック下の一瞬」に縮めて解消。*クリティカルな状態確定は、副作用 (worktree 作成) の前に
  原子的に済ませる。*
- **セッション状態の検出方式 — hook → herdr。**
  当初は Claude Code 固有 hook 由来のマーカー + `tmux capture-pane` で状態推定していたが、
  working/waiting しか区別できず alt-screen で取りこぼした。herdr の `agent_status` に信号源を移し、
  blocked/idle まで区別。*外部プロセスの状態は「画面を覗く」より「イベントを push してもらう」方が堅い。*
- **send-keys の Enter レース — 0131。**
  session-rename で `send-text` と `send-keys Enter` を 2 リクエストに分けていたため、
  プロセス起動間のギャップで Enter が「改行」として食われる稀なレースがあった。
  1 リクエストでアトミックに送る `herdr pane run` に変更。*2 手に分けた「入力+確定」は競合しうる。*
- **worktime の二重記録 → プラグインへ一本化。**
  session-hook と herdr プラグインの両方が worktime を書いて二重記録に。状態ソースを herdr に一本化。
  *同じ事実を複数経路で記録しない (single source of truth)。*
- **alloc-id の空予約ファイルが Write できない — 0122。**
  予約した空ファイルを skill が Write しようとすると Claude Code が `File has not been read yet` で弾く。
  `--title` フル生成モードで採番と同時に本文まで書き込む形にして往復を解消。
  *ツールの制約 (未読ファイルは書けない) に、データ生成側を寄せて回避した。*
- **`session:` への UUID 貼り間違い — 0130。**
  「人が開く web URL」を入れる欄に、突合用のローカル session_id (UUID) が混入しがちだった。
  doctor に URL 形式チェックを追加し、web URL を自動記録するようにして再発防止。
  *似て非なる 2 つの値 (UUID と URL) は、検査と自動化の両輪で取り違えを防ぐ。*
- **serve の自動更新がドリルダウンを巻き戻す — 0104。**
  meta refresh による全ページ再読込をやめ、JSON をポーリングしてその場で再描画する方式に。
  分析ビューでは自動更新自体をやめてスナップショット表示に (0134)。
  *「常に最新」より「操作を邪魔しない」を優先する場面がある。*
- **herdr 全面移行のリスク管理。**
  一度に切り替えず、別名ビルド + state dir 隔離で **共存させながら段階移行 → dogfood → 本採用**
  という手順を踏んだ。*大きな基盤移行は「並走できる状態」を作ってから切り替える。*

## 5. 作者の実感 (ふりかえりの対話から)

記録 (CHANGELOG / git) からは読み取れない、作者自身の実感を対話で掘り下げた。

- **狙いの答え合わせ。** 当初の狙いは 3 つ — (1) 複数エージェントを worktree で同時に回す
  **並行開発の基盤**、(2) 登録〜着手〜完了を追える**タスク管理そのもの**、(3) Claude Code / herdr を
  dogfood する**エージェント開発の実験台**。worktime / cost などの可視化は主目的というより、
  実験台として作り込んだ**副次的な成果**という位置づけ。
- **一番の発見は「記録する場所が決まる」効果。**
  「タスクを記録しておく場所が決まることで、*今何をやるべきか / 何をやれるか*を把握しやすくなった」。
  機能の派手さよりも、**散逸しがちな『やること』が 1 箇所に集まること自体**が効いた、という気づき。
  これは狙い (2) のタスク管理そのものの価値が、想定以上に大きかったことを意味する。
- **一番の摩擦は「LLM に状態遷移を安全にやらせる」こと — そしてほぼ解決した。**
  frontmatter の状態確定を LLM の手編集に委ねると壊れやすい、というのが開発中の最大の摩擦だった。
  これを `claim`/`done`/`block`/`resume` として **CLI のロックに集約**する設計で「ほぼ解決した」実感。
  §3・§4 で触れた設計判断が、作者の体感としても当たっていたことの裏付けになる。
- **今後は「エージェント以外の自分の仕事」にも広げたい。**
  機能拡張は続けつつ、agent が実行するタスクだけでなく、**自分自身の仕事の記録・ふりかえり**にも
  使えるか模索したい。具体的には (a) レビュー・返信などの作業、(b) 調査・検討・設計、
  (c) タスク以外の日報・週報。`kind: human` や `report` は既にその布石になっており、
  この方向を後押しする機能 (記録の軽量化・振り返り出力の充実など) が次の関心事。

### そのほか気になる点

- **テストの手当て。** webassets の描画ロジックは vitest が入ったが、herdr 連携・TUI・spawn 周りは
  手動確認 (dogfood) 中心。壊れやすい状態遷移 (claim/done/block/resume) は Go テストがある一方、
  herdr socket 越しの経路は自動テストが薄い。
- **ドキュメントの追随。** 機能追加のペースが速く、README / CLAUDE.md / SKILL.md の記述が
  実装に追いつききれない箇所がないか、定期的に棚卸ししたい。
- **人手タスク・code タスクの境界。** `kind: human` を導入したが、判断が曖昧なケース
  (振り返りタスクなど) の扱いは運用で詰める余地がある。§5 の「エージェント以外の仕事の記録」への
  拡張とも関わる論点。
- **エージェント以外の仕事への拡張 (§5)。** レビュー・返信、調査・検討・設計、日報・週報といった
  「自分でやる仕事」を軽く記録・ふりかえりできる形を模索したい。記録の軽量化 (着手/完了の手数を
  減らす) と、`report` を軸にした振り返り出力の充実が具体的な候補。

## 7. まとめ

12 日で 136 タスクを回し、その大半を着手当日に完了させた高速なループが、このプロジェクトの
最大の特徴。それを支えたのは初期の設計判断 (skill/CLI 分離・データを repo 外へ・agent 中立) と、
**壊れやすい状態遷移を CLI のロックに集約して LLM の手編集ミスを封じた**こと (作者の実感でも
「一番の摩擦」がこの設計で「ほぼ解決した」— §5)。最大の作り直しである herdr 全面移行は、
共存 → dogfood → 本採用という慎重な手順で乗り切った。

一方で最大の価値は派手な機能ではなく、**『やること』が 1 箇所に集まり「今何をやるべきか/やれるか」を
把握しやすくなったこと**にあった (§5)。この気づきは今後の方向 — エージェント以外の自分の仕事の記録・
ふりかえりへの拡張 — に直結する。持続的な開発の鍵は、残 todo の消化に加えて、
**速度に対して薄くなりがちなテストとドキュメントを追随させること**になる。
